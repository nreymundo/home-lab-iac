# Layered Impact Check

This repo is a four-layer provisioning chain with strict source-of-truth
boundaries. Symptoms often surface one layer downstream of the real cause.

```
Packer template  ──►  Terraform root  ──►  Ansible inventory/roles  ──►  Kubernetes/Flux
(packer/)             (terraform/)        (ansible/)                    (kubernetes/)
```

Each layer has an `AGENTS.md` that defines its source-of-truth contract. Always
read the relevant subtree `AGENTS.md` before claiming an impact assessment.

## When to use

- A change to a Packer template, Terraform module/root, Ansible role/playbook,
  or generated inventory.
- Node topology changes (labels, hostnames, users, node count).
- "Fix this Ansible failure" / "fix this Terraform plan" without checking the
  upstream source.
- Reviewing a PR that touches more than one layer.

## Inputs

- The changed path(s) (from `git diff`, a PR, or the user's description).
- Whether the change is meant to be Git-only validation or includes live ops.

## Impact matrix

Map each changed path to its layer, then walk downstream:

| Changed in | Likely downstream impact | Verify |
|---|---|---|
| `packer/<template>/` | Terraform roots that clone this template; node bring-up compatibility | `packer validate`; downstream Terraform roots that reference the template name |
| `terraform/modules/`, `terraform/instances/`, `terraform/cloud/` | Generated `ansible/inventories/*`, host labels, users, topology assumptions consumed by Ansible roles and K3s playbooks | `terraform fmt/validate/plan`; diff generated inventory; re-check Ansible grouping |
| `ansible/roles/`, `ansible/playbooks/` | K3s cluster state, host configuration, cluster assumptions Flux then builds on | `ansible-lint`, `--syntax-check`, `--check` on reachable hosts |
| `ansible/inventories/*` | Mixed-source. Files with the `DYNAMICALLY GENERATED BY TERRAFORM` header (`k3s-nodes.yml`, `openclaw.yml`, `llm.yml`, `garage.yml`, `photon.yml`, `public-vps.local.yml`) are Terraform-owned — fix upstream. The remaining host inventories (`all-vms.yml`, `baremetal.yml`, `k3s-cluster.yml`, `public-vps.yml`) and all `group_vars/` / `host_vars/` are hand-maintained — edit directly. | Check the file header before editing |
| `kubernetes/infrastructure/` | `kubernetes/clusters/production/ks/*.yaml` ordering/inclusion; SOPS secrets; services that depend on it | `kubectl kustomize kubernetes/infrastructure`; `flux get kustomizations` |
| `kubernetes/components/` | All workloads that reference the component (search before assuming local) | `kubectl kustomize` on representative consumers |
| `kubernetes/clusters/production/ks/` | Reconciliation order and rendered inventory (`prune: true` deletes removed objects) | `kubectl kustomize kubernetes/clusters/production/ks`; `flux get kustomizations` |

## Layer validation commands

Run only the ones matching changed layers. `plan`, `--check`, and live flux
reconciliation need credentials/hosts and may be live ops.

```bash
# Packer
TEMPLATE=packer/<template>
packer init "$TEMPLATE"
packer fmt -check -recursive "$TEMPLATE"
packer validate "$TEMPLATE"

# Terraform (per modified root, including cloud roots)
ROOT=terraform/instances/vm/k3s_nodes   # or terraform/instances/lxc/... or terraform/cloud/...
terraform -chdir="$ROOT" init
terraform -chdir="$ROOT" fmt -check
terraform -chdir="$ROOT" validate
terraform -chdir="$ROOT" plan    # requires backend + provider creds; live

# Ansible
ANSIBLE_CONFIG=ansible/ansible.cfg ansible-lint ansible/playbooks/ ansible/roles/
ANSIBLE_CONFIG=ansible/ansible.cfg ansible-playbook ansible/playbooks/<playbook>.yml --syntax-check
# With reachable target hosts only:
ANSIBLE_CONFIG=ansible/ansible.cfg ansible-playbook ansible/playbooks/<playbook>.yml --check

# Kubernetes
kubectl kustomize --load-restrictor=LoadRestrictionsNone <changed-path> >/dev/null
scripts/kubeconform.sh
flux get kustomizations
flux get helmreleases -A
```

## Output to produce

For each proposed change, return:

1. **Source-of-truth owner** of each changed path (which subtree `AGENTS.md`).
2. **Downstream consumers** that must be checked (concrete paths).
3. **Validation chain** to run, clearly labelled read-only vs live.
4. Any **manual live step** required that the Git change alone cannot cover
   (e.g. `terraform plan`, `ansible-playbook --check`,
   `flux reconcile kustomization flux-system --with-source`).
5. Whether the change **should be refused or redirected upstream** (e.g. a
   hand-edit to Terraform-generated inventory).

## Anti-patterns to call out

- Hand-editing a Terraform-generated file under `ansible/inventories/`
  (the ones with the `DYNAMICALLY GENERATED BY TERRAFORM` header). Note that
  `group_vars/`, `host_vars/`, and the hand-maintained host inventories are
  editable directly — check the header.
- Encoding a topology fix in Ansible when the real source is Terraform.
- Treating a Packer base-template change as local; downstream Terraform consumers
  that clone it must be re-checked.
- Treating a `kubernetes/components/` change as local; all referencing consumers
  must be inspected.
- Validating with `kubectl apply` when `kubectl --dry-run=client` or
  `flux get` answers the question.
