# Terraform Infrastructure Change

Terraform is the source of truth for VM/LXC/cloud instances and for generated
Ansible inventory. Getting module vs. root ownership, the Terraform Cloud
workspace contract, and the Bitwarden secret flow right avoids downstream
breakage in Ansible and the K3s cluster.

## When to use

- Add or change a reusable module under `terraform/modules/`.
- Add or change a concrete instance root under `terraform/instances/{vm,lxc}/`
  or `terraform/cloud/<provider>/`.
- Update a Terraform Cloud workspace binding in a root's `providers.tf`.
- Add or reshape K3s node fleet topology.
- Fix `terraform fmt`/`validate`/`plan` failures.
- Change Proxmox or Hetzner resources (compute, firewall, ssh_keys, lxc).

Pair with `layered-impact-check` whenever the change can affect generated
inventory or cluster assumptions. Use `ansible-role-playbook-change` for the
Ansible side, and stay out of `kubernetes/` unless the change requires a
manifest edit.

## Layout

```
terraform/
  modules/
    hetzner/{firewall,ssh_keys,vm}/      # reusable Hetzner building blocks
    proxmox/{lxc,vm}/                    # reusable Proxmox building blocks
    proxmox_lxc/                         # legacy combined lxc module
  instances/
    vm/<name>/     # concrete VM roots (e.g. k3s_nodes, openclaw)
    lxc/<name>/    # concrete LXC roots (e.g. garage_s3, llm, photon)
  cloud/
    hetzner/{firewall,servers,ssh_keys}/ # cloud-provider roots
```

A typical root contains: `main.tf`, `providers.tf`, `vm_definition.tf` (or
equivalent), and `ansible_inventory.tf` (when the root feeds Ansible).

## Source-of-truth boundaries

- **Modules** are reusable building blocks; **roots** are concrete deployments.
  Do not put per-instance overrides inside a module — the root owns that.
- **Root outputs and `ansible_inventory.tf`** produce the generated files under
  `ansible/inventories/` (identifiable by their
  `THIS FILE IS DYNAMICALLY GENERATED BY TERRAFORM` header: `k3s-nodes.yml`,
  `openclaw.yml`, `llm.yml`, `garage.yml`, `photon.yml`, `public-vps.local.yml`).
  Those specific files are Terraform-owned; never hand-edit them. The other
  inventories (`all-vms.yml`, `baremetal.yml`, `k3s-cluster.yml`,
  `public-vps.yml`) and all `group_vars/` / `host_vars/` are hand-maintained
  and are NOT owned by Terraform.
- **`providers.tf`** in each root module carries the Terraform Cloud workspace
  settings — treat it as part of the root-module contract.
- **K3s fleet topology**: prefer the current `local.k3s.nodes`-driven pattern
  over older count-based patterns.
- **Secrets** belong in Bitwarden via the `bitwarden-secrets` provider — not in
  `tfvars`, plaintext files, or ad hoc env handling.

## Workflow

1. Classify the change: **module** vs **root** vs **cloud root**.
2. If module-internal, check all consuming roots before editing (a module
   change is never local).
3. If adding a new instance, pick the right root location
   (`instances/vm/`, `instances/lxc/`, or `cloud/<provider>/`) and follow the
   shape of an existing sibling root.
4. If the change feeds Ansible, confirm the generated inventory diff matches
   expectations (hostnames, labels, users, topology) before relying on it.
5. Pull secrets through Bitwarden; never inline.
6. Run the validation chain below.

## Validation (per modified root)

```bash
ROOT=terraform/instances/vm/k3s_nodes   # or .../instances/lxc/<name> or .../cloud/<provider>/<root>
terraform -chdir="$ROOT" init
terraform -chdir="$ROOT" fmt -check
terraform -chdir="$ROOT" validate
terraform -chdir="$ROOT" plan    # requires backend + provider creds; live
```

`fmt -check` and `validate` are read-only and safe to run freely. `plan` and
`apply` need the configured backend and provider credentials and are live ops
— call them out explicitly and confirm before running.

Pre-commit will run `terraform-fmt`; CI does not currently run Terraform jobs,
so local validation is the gate (see `.github/workflows/ci.yml`).

## Anti-patterns

- Hand-editing any Terraform-generated file under `ansible/inventories/` (the
  ones carrying the `DYNAMICALLY GENERATED BY TERRAFORM` header) instead of
  fixing the upstream Terraform root. The hand-maintained inventories
  (`all-vms.yml`, `baremetal.yml`, `k3s-cluster.yml`, `public-vps.yml`) and
  `group_vars/` / `host_vars/` are not Terraform-owned and may be edited
  directly.
- Inlining secrets in `tfvars`, root files, or env vars when Bitwarden already
  owns that flow.
- Putting per-instance overrides inside `terraform/modules/`.
- Reintroducing count-based K3s topology instead of `local.k3s.nodes`.
- Treating `providers.tf` workspace settings as cosmetic; they are part of the
  root contract.
- Changing root outputs or inventory shape without inspecting downstream
  Ansible grouping, host labels, and users.
- Treating a Proxmox/Packer base-template change as Terraform-local; downstream
  clone compatibility must be checked.
